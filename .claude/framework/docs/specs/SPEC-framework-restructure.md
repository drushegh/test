# SPEC-framework-restructure — Framework Root Cleanup

**Status:** Draft  
**Author:** Architect (session 2026-04-24)  
**Interview answers from:** User, 2026-04-24  

---

## Problem

The framework currently litters the project root with a large `.claude/framework/` directory tree and
multiple top-level state files (TASKS.md, STATUS.md, DECISIONS.md, etc.). Consumer projects
end up with a mix of framework machinery and project code at root, making the repo feel heavy
and hard to navigate.

**User quote:** *"it's creating a massive mess in my projects because I have so many files and
folders from the framework"*

---

## Goal

Eliminate `.claude/framework/` from the project root. Move all framework scripts and state files
under `.claude/`. Only files required by external tools (Claude Code, git) remain at root.

No functionality is removed. This is a pure structural rename + path update.

---

## User Stories

| ID | As a… | I want… | So that… |
|----|-------|---------|----------|
| US1 | Consumer | `.claude/framework/` gone from my project root | My repo's root is clean and shows only my project |
| US2 | Consumer | State files (TASKS.md etc.) under `.claude/` | The project root contains only Claude Code and git essentials |
| US3 | Consumer | Automated migration on update | I don't manually move files — running apply-update handles it |
| US4 | Consumer | Post-migration integrity check | I know immediately if something broke during migration |
| US5 | Framework dev | All internal references updated | No stale `.claude/framework/` or root-path references in any framework file |

## Proposed Layout

### Root (after restructure)

```
PROJECT_ROOT/
  CLAUDE.md                     ← Claude Code required (unchanged)
  CLAUDE.framework.md           ← Claude Code auto-loads CLAUDE*.md (unchanged)
  .gitignore                    ← git required (unchanged)
  01_Project/                   ← project code (unchanged)
  .claude/                      ← all framework content moves here
```

**Decision (from interview):** CLAUDE.framework.md stays at root. Claude Code auto-loads all
`CLAUDE*.md` files from the project root. Moving it would require adopting `@`-import syntax in
CLAUDE.md and introduces version-compatibility risk.

### What needs to be built

1. **Migration script** — `.claude/framework/update/migrate-layout.sh` (moves to `.claude/framework/update/` after it runs). Handles the one-time `git mv` + reference update for any consumer.

2. **apply-update.sh migration hook** — Detects old structure (`-d 00_framework` or `TASKS.md` at root) and calls `migrate-layout.sh` automatically before applying the manifest update.

3. **Updated `framework-manifest.txt`** — All 96 paths changed from `.claude/framework/...` to `.claude/framework/...`.

4. **Updated cold start scripts** — All framework scripts with internal path references rewritten.

5. **Post-migration doctor check** — `migrate-layout.sh` runs `doctor.sh` after migration and surfaces any CRITICAL findings.

## Migration Behaviour (US3, US4)

### Trigger

When a consumer runs `bash .claude/framework/update/apply-update.sh` (new path) or the old
`bash .claude/framework/update/apply-update.sh` (still works via git mv detection):

```
apply-update.sh
  └─ detects: .claude/framework/ exists OR TASKS.md at root
  └─ calls: migrate-layout.sh
      ├─ git mv .claude/framework/ .claude/framework/
      ├─ git mv ROOT/TASKS.md .claude/TASKS.md  (etc.)
      ├─ git mv ROOT/.framework-version .claude/.framework-version
      ├─ updates CLAUDE.md references (sed in-place)
      ├─ commits: "chore: framework layout migration — .claude/framework/ → .claude/framework/"
      └─ runs: bash .claude/framework/doctor/doctor.sh
          ├─ CRITICAL findings → printed prominently, migration halted
          └─ WARN/INFO findings → printed, user proceeds
  └─ continues: normal manifest-update flow with new paths
```

### What migrate-layout.sh does NOT do

- Does not auto-fix arbitrary doctor findings — only structural path issues.
- Does not touch `01_Project/` or any consumer project files.
- Does not run the full `/healthcheck` (that's 5-15 min; user can run it manually after).
- Does not delete consumer-local files that happen to share a name with framework files.

### Backward compatibility

The old path `bash .claude/framework/update/apply-update.sh` must remain callable until the migration
runs. After migration, only `.claude/framework/update/apply-update.sh` works.

**Implementation:** ship a transitional shim at `.claude/framework/update/apply-update.sh` that
immediately re-delegates to `.claude/framework/update/apply-update.sh`. This shim is NOT
manifest-tracked; it stays on disk until the consumer deletes `.claude/framework/` (which
`migrate-layout.sh` does via `git rm -r`).

**Simpler alternative (preferred):** `migrate-layout.sh` is called automatically by the
existing `apply-update.sh` before it does anything else. Since `apply-update.sh` is the entry
point, the consumer just runs the same command they always have.

## Non-Functional Requirements

- **Atomicity:** Migration is a single git commit (or two max: mv then reference updates). No
  mid-flight broken state committed to the repo.
- **Idempotency:** Running `migrate-layout.sh` twice is safe — detects already-migrated state
  and no-ops.
- **Reversibility:** Migration commit is revertable with `git revert`. No data loss.
- **Windows compatibility:** All paths use forward slashes in scripts. `git mv` handles OS
  path differences. CR strip applied in any new `while read` loops (see GOTCHAS.md).

---

## Out of Scope

- Moving `CLAUDE.framework.md` to `.claude/` (deferred — requires @-import syntax adoption).
- Moving `01_Project/` (project code, not framework).
- Changing the content of any state files (TASKS.md etc.) — path change only.
- Full `/healthcheck` run as part of migration — recommended but not automated.
- Compacting the content of framework files (word-count reduction, token optimisation) — separate concern.
- Automated test harness for hooks (separate TASKS.md item).

## Risks & Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| sed rewrites corrupt a file (false positive match) | Medium | Grep-verify each substitution before applying; use exact anchored patterns |
| Consumer has local files at old paths (custom TASKS.md content) | Low | git mv preserves content; migration only moves, never overwrites |
| `framework-manifest.txt` has stale path after update | High | AC8 validates with a dry-run on fresh consumer before shipping |
| framework-self redirect breaks after move | Medium | AC9 explicit acceptance criterion; doctor Check 8 covers flag state |
| apply-update.sh can't find itself after path change | High | Shim or self-invocation guard handles the bootstrapping problem |
| CRLF in any new `while read` loops (Windows) | Medium | GOTCHAS.md lesson applied; CR strip in all new loops |

## Implementation Order (recommended)

Phase 1 — upstream restructure (this repo):  
1. `git mv .claude/framework/ .claude/framework/` + `git mv` all root state files  
2. Update all internal path references (automated sed sweep + manual review)  
3. Update `framework-manifest.txt`  
4. Run doctor + pattern-scan to verify AC5, AC6, AC7  
5. Single commit: `feat: framework layout migration — .claude/framework/ → .claude/framework/`

Phase 2 — consumer migration tooling:  
6. Write `migrate-layout.sh`  
7. Add migration hook to `apply-update.sh`  
8. Update MIGRATION.md with the new migration step  
9. Tag: first release with the new layout  

Phase 3 — consumer rollout:  
10. Consumer runs `apply-update.sh` → migration runs automatically → doctor passes  

**Single big-bang option:** Steps 1-5 can be done in one commit if the sed sweep is scripted
and verified. Steps 6-8 can follow in a second commit. Preferred approach per user.
